# xxvim 冷启动懒加载优化设计说明

## 目标

针对 `xxvim` 的冷启动体验做首屏优先优化：在不破坏核心编辑能力的前提下，把非首屏必需的界面增强插件延后到事件、命令或按键首次触发时加载。

## 方案选择

采用方案 B：**首屏优先懒加载**。

原因：
- 用户明确接受“首次触发某些界面功能时略慢”，适合用懒加载换取更快冷启动
- 当前配置中 UI 增强与工具面板类插件较多，集中后置可以明显减少首屏初始化负担
- 相比激进精简 dashboard / 状态栏的方案，这种方式对日常体验改动更小，风险更可控

## 架构原则

- 保持现有模块拆分：`config/plugins/*.nix` 管理插件声明，`config/lua/xxvim/*.lua` 管理运行时行为
- 优先使用 `nixvim`/lazy.nvim 风格的 `event`、`cmd`、`keys`、`ft` 触发方式懒加载
- 首屏仅保留基础编辑链路：主题、基础选项、核心补全、基础 LSP、文件打开必需能力
- 将纯展示型 UI、任务面板、测试调试面板、搜索替换面板延后初始化
- 任何被延后的插件，都需要同步检查其 `require(...)` 时机，避免在启动期被提前拉起

## 优化范围

### 1. 首批重点后置的插件
- `which-key`：改为 `VeryLazy` 或首个按键相关事件后加载
- `noice` / `notify`：保留消息能力，但避免阻塞首屏初始化
- `trouble`：改为命令或按键触发加载
- `todo-comments`、`flash`：按命令/按键首次使用时加载
- `grug-far`：通过 `XxvimSearchReplace` 命令首次使用时加载
- `overseer`、`cmake-tools`：任务系统按命令或相关键位加载
- `neotest`、`dap-ui`、`dap-virtual-text`：测试/调试工作流首次使用时加载

### 2. 需要谨慎评估的首屏 UI
- `bufferline`、`gitsigns`、`indent-blankline`、`snacks` 的部分子模块都可能带来启动成本
- 这类插件不一定全部后置，而是优先检查：
  - 是否在启动即渲染
  - 是否在空白 dashboard 场景也会初始化
  - 是否能拆成子模块分别后置
- 如果影响较大，优先延后非核心子模块，例如 `snacks.notifier`、部分 picker/explorer 扩展、非必要动画功能

### 3. 不优先动的部分
- 基础补全链路：`blink-cmp`、片段、核心 LSP capability 注入
- 基础主题与界面基线：颜色主题、核心 options、runtime path
- 项目根目录识别、基础工作流 glue code

## 风险与处理

### 提前 `require(...)` 导致失去懒加载
- 需要梳理 `extraConfigLua` 与 `config/lua/xxvim/*.lua` 中的直接 `require(...)`
- 原则是将依赖插件的 `require(...)` 放进命令回调、按键回调、插件 `config` 回调内部

### 键位失效或首次触发报错
- 需要给懒加载插件补齐 `keys` / `cmd` 触发条件
- 自定义用户命令若依赖插件，应在命令回调内再 `require(...)`，避免启动期硬依赖

### UI 体验变化
- 首次打开问题列表、测试面板、任务面板、搜索替换界面会有一次加载成本
- 这属于本轮刻意接受的取舍，重点换取冷启动首屏更快

## 验证策略

每轮调整后至少执行：
- `nix flake check`
- `nix run .#xxvim -- --headless '+qa'`

如果本轮改动涉及启动路径和 UI 插件装配，还需要额外确认：
- `nix run .#xxvim` 打开首屏无报错
- 首次触发 `trouble`、测试、调试、搜索替换等功能时能够正常自动加载

## 预期结果

- `xxvim` 冷启动首屏体感更快
- 非核心 UI 和工作流能力按需初始化
- 配置结构保持清晰，不引入大规模重构
